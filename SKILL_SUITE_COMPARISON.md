# superpowers vs spec-coexist — 批評的比較レポート

CTO / PM / SE / PG / QA の 5 サブエージェントによる独立評価をマージ。
各評価者は他評価者の結論を見ずに独立採点しているため、見解は意図的に衝突する。

---

## 0. 比較対象サマリ

| 項目 | A: obra/superpowers | B: spec-coexist (本リポジトリ) |
|---|---|---|
| スキル数 | 15 | 17 |
| ライセンス / 配布 | MIT / GitHub | リポジトリ内蔵 / 外部配布なし |
| 採用シグナル | ★141k / Fork 12k / v5.0.7 (2026-03-31) | 外部採用ゼロ |
| 保守状態 | 単独メンテ (Jesse Vincent)、活発 | 単独 (本リポジトリのみ) |
| 中核哲学 | TDD-first / 体系的プロセス / 複雑性削減 / 証拠主義 | spec-coexist (要件・基本設計を一級市民化) / RFC 2119 / 1%ルール |
| 仕様レイヤ | **無し** (brainstorm → plans から直に実装) | **有り** (problem-space → requirements → basic-design → impl) |
| サブシステム分解 | 無し (worktree は並列化のみ) | parallelizing-/finishing-subsystem-work で一級扱い |
| TDD 強制 | 同列スキル (回避容易) | implementing-from-spec の **サブスキル**、`tdd-red` / `tdd-green` 証跡ファイル |
| 起動モデル | ユーザ明示 + プロンプト内オーダリング | "1% ルール" による全メッセージ評価 |
| 名前空間 | フラット | `spec-coexist:` プレフィックス必須 |
| 共有資産 | 無し | `_shared/` (doc-lifecycle, link checker, mermaid rules, doc-reference 記法) |

両者に共通するスキル概念: `systematic-debugging`, `verification-before-completion`, `requesting/receiving-code-review`, `finishing-*-branch/subsystem`, `using-*` ブートストラップ。
**B は A の "実装〜マージ" レイヤを持ちつつ、その前段に仕様レイヤを追加している** — 同じ問題空間に対する 2 つの設計判断。

## 1. CTO 評価

### 戦略適合
- **A**: ソロシニア / 小規模プロダクトチーム向け。ブランチを越えて生き残る成果物が無い。要件もデザインもトレーサビリティも無い。SOC2 / ISO 27001 / 医療 / 金融といった規制業務では Stage 2 監査を通せないため即不採用候補。
- **B**: ギャップを正面から埋めている。RFC 2119 要件、基本設計、サブシステム分解、改訂フロー、リンク検査。複数チームを抱える組織が実際に必要とする形をしている。ただし全メッセージに発火する 1% ルールはやり過ぎで、シニア IC を消耗させる。リポジトリ単位のオプトインに後退させるべき。

### リスク
- **A**: バス係数 = 1 (Jesse Vincent)。141k スターは社会的証明であってサポート契約ではない。MIT で問題なし。v5.x までの高速反復は破壊的変更を含意。スキル本体はプロンプトなのでロックインは小。
- **B**: 外部バス係数 = 0。リリースもユーザもイシューも無い。一方で **自リポジトリに同居** しているためベンダーリスクは文字通り 0。残るリスクは品質ドリフトと作者離脱。

### コスト vs ペイオフ
- **A**: 採用コストほぼ 0、TDD・検証・デバッグ衛生で即時の見返り。ただし「1 ブランチ・1 開発者」を超えると天井が低い。
- **B**: 17 スキル + 名前空間規則 + サブスキル合成で認知負荷高。ペイオフは「実際にチームが要件・設計ドキュメントを書き続ける」前提で発生 — これは文化問題でありスキルでは解決しない。合成モデル (TDD を implementing-from-spec の下にネスト) は A のフラットなプロンプト依存より構造的に綺麗。

### スコア (/10)

| 軸 | superpowers | spec-coexist |
|---|---|---|
| 戦略価値 | **5** | **7** |
| リスクプロファイル | **5** | **6** |
| 保守性 | **6** | **5** |
| チームスケール | **4** | **7** |

8 以上は 1 つも無い。両者とも組織規模での実戦検証を欠く。

### 結論
**ハイブリッド採用**。バックボーンに A は採れない (仕様レイヤ欠如が致命)。骨格は B の仕様足場 (要件・設計・サブシステム分解・リンク検査・改訂フロー)。A から盗むのは worktree 規律、`dispatching-parallel-agents`、そして実戦で磨かれた `systematic-debugging` / `verification-before-completion` の言い回し (B 側はどう見ても派生)。**1% ルールのグローバル発火はリポジトリマーカーによるオプトインに格下げ**。A のフォークをリポジトリに pin し、上流チャーンリスクを切り離す。2 四半期後に実際の監査と複数チーム運用データで再評価。

## 2. PM 評価

### 軸別
- **要件トレーサビリティ**: A はゼロ。「plan」はブランチスコープのエンジニア手控え。要件 ID も無く、コードから出発点に遡る経路が無い。PM が「なぜこのボタンがあるのか」と聞いた答えがコミットメッセージになる。B は実際にこれをモデル化している (RFC 2119 要件、基本設計、リンク検査) — 業界的に新規ではないが、**この 2 者の間では決定的差**。
- **ステークホルダー連携**: A はエンジニア対エンジニアの構造。brainstorming も plans もステークホルダーに渡せる成果物ではない。B の要件 + 基本設計はノンエンジニアでも一応レビュー可能。RFC 2119 散文は乾いているが、doc lifecycle (draft / active / deprecated / superseded) は PM が実際に欲しいやつ。
- **スコープ / 変更管理**: A は「変更 = 新しい plan」。何がなぜ変わったかの記録が残らない。B は `revising-spec` + `revising-implementation` のペアと supersession 状態を持つ。**両者最大の差**。
- **役割間ハンドオフ**: A は「エンジニア / エージェント」という単一ロール前提。デザイナと PM に入口が無い。B は exploring-problem-space → requirements → design → implementation という流れがディスカバリ→デリバリにほぼ対応し、PM が前半を所有可能。
- **デリバリ可視性**: A は **実行内部** が強い (worktree、並列エージェント、verification-before-completion、finishing-a-branch)。B はこれらを借用しているが、独自追加 (subsystem decomposition) は未検証。

### ノンエンジニアのフリクション
A は PM にとって読めない。B は読めるが冗長 (RFC 2119 + リンク検査セレモニー)。1% ルールが全メッセージに評価を走らせるのは軽い問い合わせを苛つかせる。

### 新機能の Time-to-Value
A: 既にアイデアが固まっていれば、アイデアからマージ済 PR への最速経路。
B: コールドスタートが遅い (要件 + 設計ゲート) が、アイデアが未成熟なときの再書き換えが少ない。

### スコア (/10)

| 軸 | superpowers | spec-coexist |
|---|---|---|
| 要件管理 | **2** | **7** |
| 変更管理 | **3** | **7** |
| ステークホルダー可視性 | **2** | **6** |
| 配信スピード | **8** | **6** |
| ドキュメント品質 | **3** | **7** |

### 結論
実プロダクト組織には **spec-coexist** が適合。superpowers はメソドロジを装った単独エンジニアの速度向上ツール — 実行レイヤは見事だがプロダクト面はゼロ。spec-coexist は実戦未検証で採用ゼロが本物のリスクだが、**PM・デザイナ・エンジニアが同じ成果物を指せる唯一の選択肢**。推奨: spec-coexist の前半を採用し、superpowers の実行スキル (worktree、並列エージェント、verification、finishing-branch) を丸ごと盗む。どちらか単独では不十分。

## 3. SE / Tech Lead 評価

### アーキテクチャ健全性
- **A**: フラットな 15 スキル袋。順序はスキル本文に散文埋め込み (「brainstorming のあとに writing-plans を…」)。マニフェスト無し、依存グラフ無し、名前空間無し。結合は暗黙で脆弱 — スキル名を 1 つリネームすれば参照が黙って腐る。`writing-skills` / `using-superpowers` が構造欠如を取り繕おうとする meta スキル。スキル単体の凝集は悪くないがスイートレベルのアーキテクチャは「善意で繋いだ連結リスト」。
- **B**: レイヤード (spec → impl → review → finish)。明示的サブスキル合成 (`test-driven-implementation` ⊂ `implementing-from-spec`、`enforcing-code-discipline` ⊂ `requesting-code-review`) が実際のツリーを構成する。`_shared/` が横断関心事を集約 (doc-reference, lifecycle, link checker, mermaid)。名前空間プレフィックスで衝突防止。RFC 2119 が決定的な強制セマンティクスを与える。「実行時に superpowers に委譲してはならない」という独立性宣言は雰囲気ではなく **アーキテクチャ境界**。重いが本物の設計。

### サブシステム分解
A はゼロ。worktree を分解原語ではなく並列化原語として使用。<10kLOC のモノリスなら問題無いが、有界コンテキスト・サービス継ぎ目・独立デプロイ単位を持つ現実システムでは「1 つの巨大 plan」に潰れる。B の `parallelizing-subsystem-work` + `finishing-subsystem-work` は **実システムは継ぎ目を持ち、統合自体が仕事である** ことを認識している。継承する本番システムの大半は機能ではなく継ぎ目で破綻するため、これは大きい。

### TDD 強度
A: 同列スキル → スキップ容易、構造ゲート無し。
B: TDD は implementing-from-spec のサブスキル → ゲートを通らずに実装に到達できない。`enforcing-code-discipline` がレビュー前に第二ゲートを追加。**B は構造的、A は文化的**。

### Worktree / 並列性
A は `using-git-worktrees` + `dispatching-parallel-agents` を一級市民として持つ — 実戦的・具体的。B の `parallelizing-subsystem-work` は高水準だが worktree メカニクスについては規範性が弱い。**メカニクスは A の勝ち、いつ・なぜは B の勝ち**。

### 失敗モード
B の 1% ルール = **過剰発火**。全メッセージにスキル評価税。些細な依頼でノイズになり、ユーザを「無視する訓練」に追い込む。
A の暗黙オーダリング = **過小発火**。何もスキル走査を強制しないため、モデルは `test-driven-development` に触れずに平気でコードを書く。
**過小発火は静かに品質を劣化させ、過剰発火は煩いが観測可能** — B の失敗は声が大きく、ゆえに修正可能。

### スコア (/10)

| 軸 | A superpowers | B spec-coexist |
|---|---|---|
| アーキテクチャ | **5** | **7** |
| 合成可能性 | **4** | **7** |
| TDD 厳格性 | **5** | **7** |
| 並列作業支援 | **7** | **6** |
| 障害耐性 | **5** | **6** |

### 結論
A は単独有界コンテキストの小規模リポジトリでの単独機能開発には妥当な道具箱。フラット構造は単一コンテキストを越えてスケールしない。B は **アーキテクチャと呼ぶに値する構造** を持つ唯一の側 — 仕様レイヤ、サブスキル合成、共有原語、A への明示境界。B はそれをブートストラップノイズと重いセレモニーで支払う。実マルチサブシステム作業なら B、週末ハックなら A。両者ともどの軸でも 8 に届かない — どちらも初期段階。

## 4. PG (現役プログラマ) 評価

> コードを出すのが仕事だ。変数リネームに RFC を書く気は無い。

### 日次フリクション
- **A**: 低。brainstorm → plan → execute。plan はブランチスコープで軽量、セレモニーが作業量にスケールする。既に分かっている時は TDD や systematic-debugging に直行できる。
- **B**: 高。コード 1 行前に 4 つの成果物 (problem space、requirements、basic design、implementation)。スキルファイル本体に RFC 2119 の「MUST/SHALL」言語 — プロセス志向のシグナル。名前空間プレフィックス必須は永続的紙切れキズ。スキル数 17 vs 15 は微差に見えるが、追加 2 つは **道具ではなくゲート**。

### 小タスク (タイポ / リネーム / 10 行修正)
- **A**: 邪魔しない。plan 不要、編集して検証するだけ。これが正しいデフォルト。
- **B**: 積極的に敵対的。1% ルールが全メッセージを検査し、タイポ修正でも `exploring-problem-space` に流される可能性。たとえ迂回しても、スイートの重力が spec-first に引っ張る。戦うか無視するか — どちらも負け。

### 中タスク (機能追加)
- **A**: 構造が効くべきところに効く — plan、TDD、並列の worktree、verification、レビューループ。ブランチスコープ plan は正しい粒度。subagent-driven-dev は実際の倍速器。
- **B**: 仕様レイヤがペイするのは恐らく 20% — 真に曖昧で要件が問題になる機能。残り 80% は markdown でレンダリングされた重複思考。「basic-design 然 implementing」はウォーターフォール臭。worktree / 並列の話題が語られていないのはギャップ。

### 1% ルール
**耐え難い**。全メッセージを評価して maybe スキルを起動するインターセプタは、信頼を最速で侵食する種類の「親切な」自動化。1 行作業を spec ルートに乗せた瞬間に無効化する。理論上は秀逸、実用では税。「ユーザが明示起動」と A/B したら明示起動が勝つ。

### セルフレビュー (enforcing-code-discipline)
diff スコープのチェックリストをレビュー要求前に走らせるなら **真に有用**。アホなやつ (デバッグプリント、TODO、デッドブランチ) を捕まえる。ただし名前 — "enforcing"、"discipline" — がトーンの問題を物語る。**俺を欠陥扱いしている**。A の "verification" フレーミングは健全。同じ結果、説教は少ない。

### スコア (/10)

| 軸 | A superpowers | B spec-coexist |
|---|---|---|
| 日次エルゴノミクス | **8** | **4** |
| 小タスクのオーバーヘッド | **9** | **3** |
| 中タスクのスピード | **8** | **5** |
| 認知負荷 | **7** | **4** |
| 使う喜び | **8** | **4** |

### 結論
**A を使え**。superpowers は「ほとんどの作業は小さく、ほとんどの plan は使い捨て」を尊重している。spec-coexist は道具の姿をしたプロセス成果物 — A 内部の **オプションのサブモード** として真に spec 価値のある作業向けに配置するなら擁護できるが、デイリードライバとしてはフリクションシアター。141k スターは間違いではない。`enforcing-code-discipline` (リネームして) を盗み、残りを捨てよ。

## 5. QA 評価

### テスト規律
- **A**: RED-GREEN-REFACTOR にアンチパターンリスト — 教科書通り、しかし強制は **奨励的**。モデルに行儀よく振る舞うよう頼んでいるだけ。RED が GREEN に先行した証跡が残らない。アンチパターンはカタログ化されているがゲート化されていない。
- **B**: "Iron Law" + `tdd-red` / `tdd-green` 証跡ファイルは **物質的な前進** — 反証可能なアーティファクトシーケンスを生む。エージェントが日付を遡及しない前提に依存はするが、レビュアが grep できる。TDD を `implementing-from-spec` の下にサブスキル化することで、テストを浮遊する意図ではなく **仕様化された振る舞い** に正しく結合させている。

### 要件 → テストのトレーサビリティ
- **A**: ゼロ。要件レイヤが無いのでテストは開発者の直感にトレースする。趣味コード以外では受容不能。「REQ-X をカバーするテストはどれか」に答えられない、なぜなら REQ-X が存在しないから。
- **B**: 構造的前提条件は揃っている — 要件、基本設計、RFC 2119、リンク検査。ただし「可能」と「強制」は別。明示的な REQ-ID → test-ID マトリクススキルは見当たらない。前提インフラは存在、ラストワンマイル (双方向マトリクス、カバレッジレポート) は未実装。

### 予防 vs 検出のバランス
- **A**: 検出寄り (debugging, verification)。予防は TDD 規律のみ。仕様ゲート無し → 要件段階で生まれた欠陥はテスト時まで不可視。
- **B**: 真の予防レイヤ (spec → design → impl) + 検出は同等 (同じ debugging / verification)。シフトレフト姿勢は良い。リスクは小タスクへのセレモニー税 (スコープ制御しなければ)。

### Verification 強制可能性
両スイートとも「証拠必須」を定義。どちらも done-claim を不足証拠でブロックするサンドボックスでは走らない。A 版は純粋散文 — シアター。B 版は `tdd-red`/`tdd-green` ファイルと `check_doc_links.sh` が **検査可能なアーティファクト** を産み、`enforcing-code-discipline` がレビュー前自己チェックゲートを追加。依然自己申告だが、雰囲気ではなく **偽造可能だが監査可能なアーティファクト**。**B は意味のある度合いでシアター度が低い**。CI フック無しで真に強制可能なものは無い。

### 監査証跡 / 規制業務
- **A**: 不可。要件無し、トレーサビリティ無し、ドキュメントライフサイクル無し、適合言語無し。ISO 13485 / IEC 62304 / DO-178C の臭い嗅ぎテストを即落ちる。
- **B**: 妥当な出発点。doc lifecycle 状態、RFC 2119、リンク整合性検査、証跡ファイル — 規制 QMS が必要とする骨格。不足: 署名済みレビュー、不変履歴保証、明示的トレーサビリティマトリクス成果物、CAPA ループ。

### スコア (/10)

| 軸 | A superpowers | B spec-coexist |
|---|---|---|
| TDD 厳格性 | **6** | **7** |
| トレーサビリティ | **1** | **6** |
| 欠陥予防 | **4** | **7** |
| 検証強制可能性 | **3** | **5** |
| 監査適合 | **1** | **6** |

### 結論
superpowers は非規制ソフトウェア向けの有能な職人ツール — デバッグ衛生に強く、トレーサビリティと監査には構造的に対応不可。spec-coexist は **規制 / 仕様駆動 / 複数ステークホルダー業務に唯一候補となる側**。TDD 証跡ファイルは散文規律より具体的な改善。外部 CI ゲート無しに真の強制可能性に到達するスイートは無い。第三者に正しさを **証明** する必要があれば B、誰も「なぜ」と尋ねない領域でだけ A。

## 8. 最終勧告

両スイートを **対立物としてではなく補完物として** 扱え。

### 推奨ハイブリッド構成

1. **骨格は B (spec-coexist)** を採用する。理由: 仕様レイヤ・サブシステム分解・TDD 構造強制・監査適合骨格は、組織が後付けできない種類の能力。
2. **A から取り込むもの**:
   - `using-git-worktrees` のメカニクス本体
   - `dispatching-parallel-agents` / `subagent-driven-development` の具体的並列パターン
   - `verification-before-completion` / `systematic-debugging` の実戦化された言い回し (B 版は派生に見える — 上流の磨きを取り込む)
3. **B から修正すべきもの (致命的)**:
   - **1% ルールのグローバル発火を廃止**。リポジトリマーカー or 明示オプトインに格下げ。これを直さなければ評価者 5 名中 4 名が指摘した「過剰発火税」が組織採用の最大障壁となる。
   - `enforcing-code-discipline` のトーンを `verification` 系に揃える (PG 評価)。
   - REQ-ID → test-ID トレーサビリティマトリクスを生成する明示的スキルを追加する (QA 評価のラストワンマイル)。
4. **B の構造的弱点**: 外部採用ゼロ・実戦未検証。**A の関連スキルを fork して `_shared/` 配下に pin** し、上流チャーンと自前メンテ両方のリスクを下げる。
5. **再評価ゲート**: 採用後 2 四半期、(a) 実監査を 1 件通す、(b) ≥2 チームで運用、を満たした時点で本レポートを再採点。

### コンテキスト別推奨

| コンテキスト | 推奨 |
|---|---|
| 規制業務 (医療 / 金融 / 安全クリティカル) | **B 必須**。A では監査落ち。 |
| マルチチーム / マルチサブシステム製品 | **B + A の実行レイヤ移植** |
| ソロ開発 / プロトタイピング / 内製ツール | **A**。B は過剰。 |
| OSS / ホビー | **A**。141k スターには理由がある。 |
| 公的 / 第三者証明可能性が要件 | **B のみ**。A は構造的に不適。 |
| デイリードライバとしての「速さ」最優先 | **A**。 |

### 一行結論

> **superpowers は速度を、spec-coexist は証明可能性を解く。プロフェッショナル現場の大半は両方必要としており、そのどちらかだけを選ぶことはアーキテクチャ的怠慢である。**
