# 美しい Mermaid State Diagram 作成ルール

本ドキュメントは、日本語の設計ドキュメント（要件定義書・基本設計書など）において、Mermaid の **State Diagram (stateDiagram-v2)** を「読みやすく」「スケールしても破綻しない」形で記述するための原則をまとめたものです。

---

## 1. 概要と用途

State Diagram は、対象（エンティティ・画面・ジョブ・セッションなど）の **ライフサイクル** と **イベント駆動の振る舞い** を表現する図です。UML ステートマシン図に対応します。

主な用途:

- 業務エンティティ（注文、申請、契約 等）の状態遷移
- 画面・ウィザードのナビゲーション
- 非同期ジョブ・バッチ処理のステータス管理
- プロトコル・接続セッションの状態管理
- 例外・リトライを含むエラーハンドリングの可視化

**使うべきでない場面:**

- 単なる処理フロー（→ Flowchart を使う）
- 主体と相手のメッセージ交換（→ Sequence Diagram を使う）
- データ構造の関係（→ ER / Class Diagram を使う）

> 原則: 「主語が一つ」で「状態を持ち」「イベントで遷移する」場合のみ State Diagram を使う。

---

## 2. `[*]` 開始 / 終了の扱い

`[*]` は擬似状態であり、開始 (initial) と終了 (final) の双方を表します。矢印の向きで意味が決まります。

- `[*] --> Draft` … 開始
- `Closed --> [*]` … 終了

**ルール:**

1. すべてのトップレベル図に **必ず開始 `[*]` を記述する**（暗黙の開始を避ける）。
2. 終了が複数あるとき（正常終了 / キャンセル / 失敗）は、`[*]` を 1 つに集約せず、意味に応じて分けて書くか、明示的な終了状態を設けてからまとめて `[*]` に流す。
3. 複合状態の内部にも、その内部だけの `[*]` を置いてよい（その複合状態に入った時の初期サブ状態を示す）。

---

## 3. 状態の命名規則

状態は「現在その対象がどういう状況にあるか」を **名詞句または形容詞句** で表します。動詞句（処理名）を入れない。

| 良い例 | 悪い例 | 理由 |
|---|---|---|
| `審査待ち` | `審査する` | 動詞は遷移／アクション |
| `配送中` | `配送処理` | 「処理」は工程名で曖昧 |
| `支払済` | `支払完了処理` | 状態は結果としての様態 |
| `下書き` | `編集画面` | 画面名と状態を混同しない |

**追加ルール:**

- 同じ図内で粒度を揃える（「申込」と「与信審査中の中断状態」のような粒度差を避ける）。
- 英語と日本語を混在させない。日本語ドキュメントなら状態名は日本語で統一。
- ID には空白や記号を避け、表示名は `state "下書き" as Draft` 形式で別名定義する。

```
stateDiagram-v2
  state "下書き" as Draft
  state "審査待ち" as PendingReview
  Draft --> PendingReview: 提出
```

---

## 4. 遷移ラベル

UML の標準形式 **`イベント [ガード] / アクション`** に従います。Mermaid ではコロン `:` の後にこの文字列を書きます。

```
Draft --> PendingReview: 提出 [入力完了] / 通知メール送信
```

**ルール:**

1. 1 つの遷移に複数のイベントを詰め込まない。違うイベントなら矢印を分ける。
2. ガード条件は角括弧 `[ ]` で短く書く。長い条件は `note` で補足する。
3. アクションは副作用（外部通知、ログ、状態変数の更新）に限り、内部処理の手順は書かない。
4. ラベル無しの遷移（自動遷移）は理由を必ず note で示す。

---

## 5. 複合状態 (composite state) とネスト深さ

関連する状態をまとめるには複合状態を用います。

```
stateDiagram-v2
  [*] --> 受付中
  state 受付中 {
    [*] --> 入力中
    入力中 --> 確認中: 次へ
    確認中 --> 入力中: 戻る
  }
  受付中 --> 完了: 確定
  完了 --> [*]
```

**ネスト深さの上限: 原則 2 階層まで（最大 3）。**
それ以上深くしたい場合は、子の複合状態を **別図（サブマシン）に切り出す**。深いネストは可読性とレンダリング品質を著しく損ないます。

---

## 6. 並行状態 (concurrent regions)

1 つの対象が同時に複数の独立した状態を持つ場合、複合状態の中で `--` により領域を区切ります。

```
stateDiagram-v2
  state 通話中 {
    [*] --> 音声ON
    音声ON --> 音声OFF: ミュート
    音声OFF --> 音声ON: 解除
    --
    [*] --> 映像ON
    映像ON --> 映像OFF: カメラOFF
    映像OFF --> 映像ON: カメラON
  }
```

**ルール:**

- 並行領域は **2〜3 個まで**。それ以上は別図に分ける。
- 領域間の依存（一方の状態が他方に影響する）は **note で明示**。図の上では交差させない。

---

## 7. choice / fork / join ノード

- `<<choice>>`: ガード条件による分岐（菱形）
- `<<fork>>`: 1 → 多 の並行開始
- `<<join>>`: 多 → 1 の同期

```
stateDiagram-v2
  state if_state <<choice>>
  受付 --> if_state: 申請受領
  if_state --> 自動承認: [金額 < 1万円]
  if_state --> 人手審査: [金額 >= 1万円]
```

```
stateDiagram-v2
  state fork_state <<fork>>
  state join_state <<join>>
  開始 --> fork_state
  fork_state --> 在庫確保
  fork_state --> 与信確認
  在庫確保 --> join_state
  与信確認 --> join_state
  join_state --> 出荷準備
```

choice は **2〜3 分岐まで**。4 つ以上になる場合は状態の設計を見直す。

---

## 8. direction TB / LR の使い分け

```
stateDiagram-v2
  direction LR
  ...
```

- **LR（左→右）**: 工程的・時系列的なライフサイクル（申請 → 審査 → 承認 → 完了）
- **TB（上→下）**: 階層的・分類的、または並行領域が多い図
- 複合状態の内部にも `direction` を指定でき、外側と独立して制御できる
- 1 つの図で原則 1 種類に統一。内側だけ向きを変える場合は意味があるときに限る。

---

## 9. note の活用

```
stateDiagram-v2
  審査中 --> 承認済: 承認
  note right of 審査中
    SLA: 受付から 2 営業日以内
    担当: 与信チーム
  end note
```

**用途:**

- 長いガード条件・ビジネスルールの補足
- SLA・タイムアウト値・関係者の明示
- 外部システム連携の説明

note は **状態あたり 1 個まで**。多くなる場合は本文ドキュメント側に逃がす。

---

## 10. 大規模化への対処

状態数が増えると state diagram は急速に読めなくなります。以下の方針で抑えます。

1. **階層化**: 関連する状態は複合状態にまとめる。最初から平らに並べない。
2. **サブマシンへの分割**: 複合状態の中身は別図にし、本図ではブラックボックスとして 1 状態で扱う（図の見出しでリンクを示す）。
3. **観点ごとに図を分ける**: 「正常系のみの図」「エラー・リトライ系の図」「並行リソースの図」のように、観点で図を分ける。
4. **状態数の上限目安**: 1 図あたり **15 状態以内**。超えたら必ず分割を検討する。
5. **遷移数の上限目安**: 1 図あたり **30 遷移以内**。

## 11. アンチパターン

### 11.1 状態爆発

属性の組み合わせをすべて状態化してしまう。

**対策**: 直交する属性は **並行領域** に分ける。または属性は状態ではなくデータとして扱い、ガード条件で扱う。

### 11.2 遷移の交差多数

「どこからでもキャンセルできる」を全て個別の矢印で書く。

**対策**: 共通の祖先となる複合状態を作り、複合状態 → キャンセル の 1 本の遷移に集約する（UML の "transition from composite state"）。

### 11.3 意味の重複する状態

`保留` と `中断` と `待機` のような近接した状態を併存させる。

**対策**: 用語集を作り、1 意味 1 状態名に正規化する。

### 11.4 動詞名の状態

`登録する` `送信する` のような状態名。状態は様態であって処理ではない。

### 11.5 終了の欠落

最終状態 `[*]` が無く、対象がいつ終わるか不明。

## 12. Good / Bad の具体例

### 12.1 Bad: 平らで動詞混じり、開始終了なし

```
stateDiagram-v2
  下書き作成 --> 申請する
  申請する --> 審査する
  審査する --> 承認する
  審査する --> 差戻す
  差戻す --> 下書き作成
  承認する --> 完了する
```

問題: 状態名が動詞、`[*]` が無い、差戻し時のガードや原因が不明。

### 12.2 Good: 名詞句、開始終了あり、ガードとアクション

```
stateDiagram-v2
  direction LR
  [*] --> 下書き
  下書き --> 審査待ち: 提出 [必須項目入力済]
  審査待ち --> 承認済: 承認 / 通知送信
  審査待ち --> 差戻し中: 差戻し [指摘あり] / コメント保存
  差戻し中 --> 下書き: 修正開始
  承認済 --> [*]
```

### 12.3 Bad: ネストが深すぎる

```
stateDiagram-v2
  state A {
    state B {
      state C {
        state D {
          [*] --> E
        }
      }
    }
  }
```

### 12.4 Good: 2 階層に抑え、深い部分は別図に分割

```
stateDiagram-v2
  [*] --> 受付
  state 受付 {
    [*] --> 入力
    入力 --> 確認
    確認 --> 入力: 戻る
  }
  受付 --> 審査: 確定
  審査 --> [*]
  note right of 審査
    詳細は「審査サブマシン図」を参照
  end note
```

### 12.5 Bad: 全状態からキャンセルへ矢印

```
stateDiagram-v2
  下書き --> キャンセル済: 取消
  審査待ち --> キャンセル済: 取消
  差戻し中 --> キャンセル済: 取消
  承認済 --> キャンセル済: 取消
```

### 12.6 Good: 複合状態でまとめてキャンセル

```
stateDiagram-v2
  [*] --> 進行中
  state 進行中 {
    [*] --> 下書き
    下書き --> 審査待ち: 提出
    審査待ち --> 差戻し中: 差戻し
    差戻し中 --> 下書き: 修正
    審査待ち --> 承認済: 承認
  }
  進行中 --> キャンセル済: 取消 [取消可能期間内]
  承認済 --> [*]
  キャンセル済 --> [*]
```

### 12.7 Good: 並行領域の活用

```
stateDiagram-v2
  [*] --> 会議中
  state 会議中 {
    [*] --> マイクOn
    マイクOn --> マイクOff: ミュート
    マイクOff --> マイクOn: 解除
    --
    [*] --> カメラOn
    カメラOn --> カメラOff: 停止
    カメラOff --> カメラOn: 開始
  }
  会議中 --> [*]: 退出
```

### 12.8 Good: choice による条件分岐

```
stateDiagram-v2
  state 金額判定 <<choice>>
  [*] --> 申請受領
  申請受領 --> 金額判定
  金額判定 --> 自動承認: [金額 < 10000]
  金額判定 --> 人手審査: [金額 >= 10000]
  自動承認 --> [*]
  人手審査 --> [*]
```

## 13. チェックリスト

レビュー時、以下を満たしているか確認してください。

- [ ] 開始 `[*]` と終了 `[*]` が明示されている
- [ ] 状態名は名詞句で、粒度が揃っている
- [ ] 遷移ラベルは `イベント [ガード] / アクション` 形式
- [ ] ネスト深さは 2 階層以内
- [ ] 1 図 15 状態 / 30 遷移以内
- [ ] 並行領域は 3 個以内
- [ ] choice 分岐は 3 個以内
- [ ] 同じ意味の状態が重複していない
- [ ] 共通の遷移（キャンセル等）は複合状態でまとめている
- [ ] direction が図の意図に合っている (LR=時系列 / TB=階層)
- [ ] note は補足に限定し、図あたり過剰に置かれていない
