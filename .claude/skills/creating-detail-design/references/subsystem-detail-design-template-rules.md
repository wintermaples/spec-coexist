# subsystem-detail-design-template.md 記述ルール

本書は `detail-design/index.md`（サブシステム単位の詳細設計書）および各モジュールファイルの執筆ガイドである。

## 全体方針

- 対象は **ひとつのサブシステム** に閉じた詳細設計のみ。プロジェクト全体共通のパターンは `main-detail-design/index.md` に従う。
- **Mermaid 図が主、コードは例外。** コードは Mermaid 図では実装のブレを防げない場合のみ許可する。
- **実装レベルには踏み込まない。** 振る舞いと契約を定義し、関数の実装や内部データ構造の詳細は書かない。
- 粒度の目安：**「実装者がこの文書を読めば、設計判断に迷わずコードを書ける」** レベル。

---

## 記法に関するポリシー — 必読

### Mermaid 図の使い分け

| 表現したいこと | 推奨する Mermaid 図 |
| --- | --- |
| モジュール間の呼出順序 | `sequenceDiagram` |
| エンティティ・コンポーネントの状態遷移 | `stateDiagram-v2` |
| 処理の分岐・判定ロジック | `flowchart` |
| データ構造・インターフェース契約 | `classDiagram` |
| データの流れ・変換 | `flowchart`（データフロー図として） |
| モジュール依存関係 | `flowchart` |
| エンティティ間の関連 | `erDiagram` |

### コードを許可する場面（限定列挙）

以下に該当する場合のみ、コードスニペットの使用を許可する：

1. **バリデーション正規表現** — 図では表現不可能な文字列パターン
2. **シリアライズ形式** — JSON Schema, Protocol Buffers 定義等の厳密なフォーマット
3. **暗号化・ハッシュパラメータ** — アルゴリズム選択・鍵長等の設定値
4. **設定ファイルのスキーマ** — YAML/TOML 等の構造定義

コードを使用する場合：
- `<!-- コード理由: Mermaid では正規表現パターンを表現できないため -->` のようにコメントを付ける
- 実装コード（関数本体、メソッド実装）は**絶対に書かない**

## 分量に関するポリシー — 必読

- **目安行数・想定ページ数は存在しない。**
- **質 > 量。** 章ごとの目的に必要十分な Mermaid 図と表で書く。
- **不要な章・セクションは「該当なし — 理由：…」で確定し、見出しは残す。**
- **テンプレを埋めるための作文を禁じる。**
- 1 モジュールの詳細設計が 30〜50 行で済むのは正しい。複雑なモジュールが 200 行になるのも正しい。**モジュールの複雑さに対して文書量がスケールしている** かを書き終わりに自問する。

## index.md の各章

### 1. 本書の目的
読者と用途、基本設計書との関係を明記。

### 2. モジュール構成

#### 2.1 モジュール一覧
基本設計書 §4 アプリケーション機能設計の機能を、凝集度の高いグループにまとめてモジュール化する。各モジュールにモジュールIDを付与し、対応する詳細設計ファイルへのリンクを記載する。

#### 2.2 モジュール依存関係図
`flowchart` でモジュール間の依存方向を示す。循環依存があれば設計の見直しを検討する。

#### 2.3 基本設計との対応
基本設計の機能ID / DES-ID とモジュールIDの対応表。**全ての DES-ID が少なくとも 1 つのモジュールに対応していること**を確認する。

### 3. 横断的設計事項
このサブシステム内で複数モジュールが共有するパターン（エラー処理、データ変換、認証フロー等）を Mermaid 図で定義する。共有パターンが無い場合は「該当なし — 理由：…」で確定。

### 4. 設計判断記録
このサブシステム内での設計判断を ADR スタイルで記録する。「何を」「なぜ」「どの選択肢から」選んだかを残す。

### 5. 課題・リスク
サブシステム内に閉じる設計課題のみ。

## モジュールファイルの各章

各モジュールファイルは `references/module-template.md` に従う。

### 1. 責務と境界
- **1.1 責務**: 1〜3 文で「このモジュールは何をするか」。
- **1.2 しないこと**: 隣接モジュールとの境界を明確化。これにより実装者が「どこまで作ればいいか」を判断できる。

### 2. 振る舞い設計
- **2.1 主要シーケンス**: `sequenceDiagram` で正常系の呼出順序を示す。外部アクター・他モジュール・データストアとのインタラクションを含む。
- **2.2 代替・異常シーケンス**: 入力不正、タイムアウト、依存先エラー等の異常系を `sequenceDiagram` で示す。正常系との差分が明確になるように描く。
- **2.3 状態遷移**: エンティティやセッションが複数の状態を持つ場合、`stateDiagram-v2` で定義。ステートレスなモジュールは「該当なし — 理由：ステートレスな処理のみ」で確定。

### 3. インターフェース定義
- **3.1 入力**: このモジュールが受け取るデータの契約を表で定義。
- **3.2 出力**: このモジュールが返すデータの契約を表で定義。
- **3.3 データ契約**: `classDiagram` で入出力の DTO 構造を図示。フィールド名・型・制約を含む。

### 4. 処理フロー
- **4.1 メイン処理フロー**: `flowchart` でメインの処理フローを示す。分岐・ループ・外部呼出を含む。
- **4.2 判定ロジック**: 条件分岐が複雑な場合、判定表（デシジョンテーブル）で整理する。Mermaid `flowchart` と併用可。

### 5. エラー処理
- **5.1 エラー分類**: エラーコード、種別、発生条件、対応（リトライ/通知/ログのみ等）、通知先を表で定義。
- **5.2 リカバリフロー**: リカバリが必要な場合、`flowchart` でリカバリ手順を示す。

### 6. 依存関係
依存先モジュール・外部サービスと、各依存先が障害時にどう振る舞うか（フォールバック、サーキットブレーカー、エラー伝搬等）を表で定義。
