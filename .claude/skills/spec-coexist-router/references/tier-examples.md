# Tier Examples — ティア判定の具体例

各ティアにつき 10 例以上を示す。判断に迷ったときの基準として参照する。

## T0: trivial (直接編集、プロセス不要)

1. README.md の typo 修正
2. 変数名を camelCase → snake_case にリネーム
3. 未使用 import の削除
4. コメントの誤字修正
5. `.gitignore` にエントリ追加
6. 設定ファイルのポート番号変更
7. ログメッセージの文言修正
8. CSS の色コード変更
9. package.json の description 更新
10. lint 警告の修正 (ロジック変更なし)
11. テストの `describe` ブロック名修正
12. 環境変数名のリネーム (1 箇所)

## T1: small (TDD + verification のみ)

1. 既存モジュールに 1 関数追加 (例: `formatDate()`)
2. 単一バグ修正 (off-by-one エラー)
3. 既存関数にバリデーション追加
4. 既存テストに境界値ケース追加
5. API レスポンスにフィールド 1 つ追加
6. エラーメッセージの改善 (ロジック変更あり)
7. 既存コンポーネントに props 1 つ追加
8. 小さなリファクタ (extract method、1 関数分割)
9. 既存 enum に値 1 つ追加 + ハンドラ追加
10. 型定義の修正 (optional → required 等)
11. レート制限の閾値変更 (テスト必要)
12. DB クエリの WHERE 条件修正

## T2: medium (implementing-from-spec light mode)

1. 新しい API エンドポイント追加 (CRUD の 1 つ)
2. 既存画面にフォーム追加
3. 既存サブシステムにキャッシュ層追加
4. 認証ミドルウェアの既存フローに 2FA ステップ追加
5. CSV エクスポート機能追加
6. 既存テーブルにカラム追加 + マイグレーション + コード修正
7. Webhook 受信エンドポイント追加
8. 既存バッチ処理にリトライロジック追加
9. 既存コンポーネントのレスポンシブ対応
10. ロギング基盤の既存サブシステムへの組み込み

## T3: large (フルパイプライン)

1. 新サブシステム (通知サービス) の設計〜実装
2. REST API → GraphQL への移行
3. モノリス → マイクロサービス分割
4. 認証基盤の全面刷新 (セッション → JWT)
5. リアルタイム機能追加 (WebSocket 基盤)
6. マルチテナント対応
7. 決済システム統合
8. データパイプライン構築
9. 管理画面の新規構築
10. 国際化 (i18n) 基盤導入

## 判定に迷うケース

| ケース | 判定 | 理由 |
|---|---|---|
| "typo 修正" と言いつつ変数名が 5 ファイルにまたがる | T1 | 変更ファイル数 > 2、テストで確認すべき |
| バグ修正だが原因が 2 サブシステムにまたがる | T2–T3 | サブシステム横断 |
| "リファクタ" だが public API は変わらない | T1–T2 | 行数と影響範囲で判断 |
| 新機能だが 10 行で済む | T1 | 行数が少ない、振る舞い変更も軽微 |
| 設定変更だがデプロイパイプラインに影響 | T2 | 振る舞い変更あり |
