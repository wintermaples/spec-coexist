# subsystem-basic-design-template.md 記述ルール

本書は `subsystem-basic-design-template.md`（サブシステム単位の基本設計書）の各章・各項に「何を書くべきか」を定義する執筆ガイドである。IPA『機能要件の合意形成ガイド』、若手エンジニアの羅針盤、NotePM の基本設計書ガイドの観点を反映している。

## 全体方針

- 対象は **ひとつのサブシステム** に閉じた設計のみ。プロジェクト全体で共通の方針・規約は `main-basic-design-template.md` に従い、本書では **差分と具体化** のみを書く。
- 粒度の目安：**「詳細設計者が迷わず実装に落とせる」かつ「要件との対応が追跡できる」** レベル。
- 全成果物に対応する要件 ID・機能 ID を必ず記載し、要件 ⇔ 設計のトレーサビリティを確保する。

---

## 分量に関するポリシー (length policy) — 必読

**本書に「目安行数」「想定ページ数」は存在しない。** 過去の出力長（例：約 1000 行）に揃えにいくことを禁ずる。

- **質 > 量。** 章ごとの目的に必要十分な情報量で書く。短く済む章は短く、深く書くべき章は深く書く。
- **書くことが無いのに肉付けしない。** 「もう少し厚みが要る気がする」は過剰加筆のサイン。代わりに「該当なし — 理由：…」または「main 側方針に従う」を 1 行で確定させる。
- **不要な章は「該当なし — 理由：…」で確定し、見出しは残す。** テンプレートの章を「埋めるべきノルマ」と見做して無理に作文するのが最悪の失敗モード。該当性が低いと感じた瞬間に N/A で確定させる。
  - 例：画面のみのサブシステムの「7. バッチ設計」→「該当なし — 理由：本サブシステムは同期処理のみで非同期バッチを持たない」と 1 行で確定する。帳票設計・外部インターフェース設計・移行設計などの章でも同様の判断を行う。
  - **見出し自体は絶対に削除しない。** テンプレート由来の見出し（章・節・項）はすべて残す。見出しを残すことで、後続の読者・レビュアーが「論点を検討して N/A と判断したのか」「そもそも未検討なのか」を区別できる。
  - 判断は画面設計 §5 だけでなく、**全章に同じく適用する**。「もう少し書いた方が網羅的に見える」と感じた瞬間こそ N/A を選ぶサイン。
  - verification gate は `TBD` / `TODO` / `???` / 空の箇条書きなどプレースホルダ残りを fail する。「該当なし — 理由：…」で 1 行確定していれば fail しない。
- **規模に応じて自然に増減する。** 1 機能のサブシステム設計が 50〜100 行で済むのは正しい。20 機能あるサブシステムが 200 行で収まることはない。書き終わりに、**サブシステムの実体に対して文書量がスケールしている**かを自問する。
- **テンプレを埋めるための作文を禁ずる。** テンプレートは「論点の網羅チェックリスト」であり、各見出しを長文で埋める課題ではない。

## 5. 画面設計

### 章全体の方針

- **Markdown でレイアウトを精密描写しない。** 本章は「画面の構成要素 (components) と振る舞い (behaviors)」を文章・表で確定させることに集中する。ピクセル位置・余白・フォントサイズ等の見た目はワイヤーフレーム画像や Figma などにオフロードし、本章からはリンクで参照する。
- **実装者が参照したときに、このサブシステムの全画面に対して以下が** すべて拾えること **を到達点とする：**
  - (a) どんな UI 部品が並ぶか
  - (b) どの項目を入出力するか
  - (c) どのイベントで何が起きるか
  - (d) どんな状態を取り得るか
  - (e) 誰が見える／触れるか
  - (f) どんなメッセージが出るか
- 「該当しない」項目は **空欄で残さず**、「N/A — 理由：…」を 1 行で書く。verification gate が空セクションを fail する。

### 5.1 画面一覧
画面ID／画面名／概要／利用者ロール／対応機能ID。要件定義書の機能一覧と 1:1〜1:N で対応させる。

### 5.2 画面遷移図
mermaid `stateDiagram-v2` または `flowchart` で**正常系と例外系**（認証失敗、権限不足、タイムアウト、未保存変更あり等）をまとめて描く。

### 5.3 共通UI方針
全体デザインシステム（main 側）からの**差分のみ**を書く。差分がなければ「main 側方針に従う」と 1 行で記載。

### 5.4 画面定義 (画面ごとの必須項目)

各画面について `a)`〜`k)` を順に埋める。a)〜j) は原則必須、k) は任意。

#### a) 概要
目的・利用シーン・対応機能ID・利用者ロール。要件側の機能ID と紐付ける。

#### b) 画面構成 (領域分割)
画面を**意味的な領域**に分割し列挙する（例: ヘッダ／グローバルナビ／検索フォーム／結果一覧／詳細プレビュー／操作パネル／フッタ）。**ピクセル位置ではなく論理ブロック**を書く。各領域に含まれるコンポーネントID を c) と紐付ける。

#### c) コンポーネント一覧
表示・入力・操作に使う UI 部品を漏れなく列挙する。共通コンポーネントは名前のみ、カスタム部品は概要も書く。フォーム／テーブル／ボタン／モーダル／トースト／タブ／アコーディオン／ツールチップ等の**種別**を必ず明記する。

#### d) 入出力項目定義
入力フォームの入力項目と、表示専用フィールドを**まとめて**列挙する。各項目に: 型・桁・必須・初期値・入力チェック（範囲・文字種・正規表現）・**表示条件**（特定状態でのみ見える等）・**編集条件**（disabled / read-only の発火条件）・備考。サーバ側で再検証する旨も備考に書く（クライアント検証だけに依存しない）。

#### e) アクション・イベント
ボタン押下／リンククリック／submit／onBlur／onChange／キーボードショートカット等、**契機イベント**ごとに 1 行。各行に: 前提条件・クライアント側検証・処理概要（呼出 API／関数名）・成功時挙動（遷移／更新／メッセージ）・失敗時挙動。**「アクション → 結果」の対応が画面上で曖昧に残らないこと**を確認する。

#### f) 画面状態
画面が取り得る状態を網羅する。最低限: 初期表示・ローディング中・データなし (empty)・エラー・完了/成功。各状態で「どの領域・コンポーネントが、どう見えるか」の差分を書く。状態遷移が複雑なら mermaid `stateDiagram-v2` を併記して良い。

#### g) 権限・可視性 (アクセス制御)
利用者ロール × 「閲覧可否／編集可否／操作可能アクション」のマトリクスで定義する。要件側の認可ルールに追跡可能にする。

#### h) メッセージ一覧
情報／警告／エラー／確認 ダイアログの**全文言**をテーブルで管理する。表示契機・表示位置（トースト／モーダル／インラインバリデーション）・対応するアクションIDを書く。これにより i18n／文言レビューを後工程で一括できる。

#### i) アクセシビリティ要件
キーボード操作・タブ順序・フォーカス制御（モーダル開閉時の戻り先等）・スクリーンリーダー向け ARIA 属性・代替テキスト・カラーコントラスト・フォントサイズ。WCAG 等の準拠基準は main 側で宣言済みの場合は本書では差分のみ。

#### j) 画面間連携 (パラメータ受け渡し)
遷移元から受け取るパラメータ／遷移先へ渡すパラメータ／「戻る」「キャンセル」「未保存変更あり」時の挙動。5.2 の遷移図と矛盾しないこと。

#### k) レイアウト概念図 (任意)
ワイヤーフレーム画像／Figma URL／Mermaid `block-beta` 等。**本書ファイル内に ASCII アートを描かない**。成果物は **付録 C 画面モックアップ集** に置き、本セクションからはリンクのみ。

## 9. データベース設計
### 9.1 ER図
機能実現に必要なテーブルを網羅的に書き出す。正規化の例外（意図的な非正規化）は理由をコメントで残す。

### 9.2 テーブル一覧
テーブルID／論理名／物理名／概要／保存期間／想定件数（初期／増加／5年後）。

### 9.3 テーブル定義
列名／物理名／型／桁／NULL／PK／FK／一意制約／初期値／備考。インデックス設計（主要検索パターン）も併記。

### 9.4 CRUD図
機能×テーブルでC/R/U/Dを埋める。**ここで埋まらないセルは機能漏れまたはテーブル漏れの兆候**。

### 9.5 データ量見積・保存期間
法令・業務要求に基づく保存期間と、パージ／アーカイブ方針。

※ 全体基本設計書（main）を継承し、**差分および具体化**のみ書く。差分なしなら「main側に従う」と記載。

### 10.1 可用性設計

### 10.4 運用・保守設計
監視項目、アラート閾値、ログ設計、ジョブ運用、定常作業一覧。

### 10.5 移行設計

## 12. テスト方針
### 12.0 テスト戦略 tier (test-strategy) — **必須**
本サブシステムの TDD 実施単位を決定する **必須** 項目。`strict` / `pipeline` / `ui` のいずれかを明記し、選定理由を 1〜3 文で述べる。未指定は `strict` 既定だが、画面 (§5) を伴うサブシステムやバッチ (§7) を伴うサブシステムで省略すると `spec-coexist:implementing-from-spec` は HALT する。tier は `spec-coexist:implementing-from-spec` / `spec-coexist:revising` / `verification-before-completion` が参照し、RED 証跡の粒度 (criterion / stage / interaction) を決める。詳細は `.claude/skills/implementing-from-spec/references/tdd-discipline.md` §Test Strategy Tiers。

### 12.1 テスト種別
本サブシステムで実施する単体／結合／総合／受入の観点、合格基準、必要な性能・セキュリティテスト。全体方針との差分を明示。
